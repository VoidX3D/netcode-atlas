Frequently Asked Questions

Find answers to common questions about web technologies and internet protocols

DNS Questions

What is DNS and how does it work? +

DNS (Domain Name System) is the phonebook of the internet. It translates human-readable domain names (like netcode-atlas.dev) into machine-readable IP addresses (like 192.0.2.1). When you enter a URL in your browser, it queries DNS servers to find the corresponding IP address before establishing a connection to the website.

What is the difference between authoritative and recursive DNS servers? +

Authoritative DNS servers store the actual DNS records for specific domains. They provide definitive answers to DNS queries for those domains.

Recursive DNS servers (or resolvers) act as intermediaries between clients and authoritative servers. They receive DNS queries from clients, perform the necessary lookups across multiple DNS servers, and return the final answer to the client.

What is DNS caching and how does it improve performance? +

DNS caching stores DNS query results temporarily to improve performance and reduce load on DNS servers. When a DNS resolver receives a query, it first checks its cache. If the answer exists and hasn't expired (based on TTL values), it returns the cached result instead of querying authoritative servers again.

Caching occurs at multiple levels: operating system, browser, ISP resolvers, and other intermediate DNS servers.

What are DNS record types and their purposes? +

Common DNS record types include:

  • A Record - Maps a domain to an IPv4 address
  • AAAA Record - Maps a domain to an IPv6 address
  • CNAME Record - Creates an alias from one domain to another
  • MX Record - Specifies mail servers for a domain
  • TXT Record - Stores text information, often for verification
  • NS Record - Specifies authoritative name servers
  • SOA Record - Stores administrative information about a zone
What is DNSSEC and why is it important? +

DNSSEC (DNS Security Extensions) adds security to the Domain Name System by enabling DNS responses to be validated cryptographically. It protects against attacks like DNS cache poisoning and spoofing by providing:

  • Data origin authentication - Verifying that data came from the correct source
  • Data integrity protection - Ensuring data wasn't modified in transit

DNSSEC uses public-key cryptography to create digital signatures for DNS data, allowing resolvers to verify the authenticity of DNS responses.

Web Technology Questions

What is the difference between HTTP/1.1, HTTP/2, and HTTP/3? +

HTTP/1.1 (1997): Uses multiple TCP connections for parallel requests, has header redundancy, and no prioritization.

HTTP/2 (2015): Introduces multiplexing (multiple requests over a single connection), header compression, server push, and request prioritization.

HTTP/3 (2022): Uses QUIC protocol over UDP instead of TCP, reducing connection establishment latency and improving performance especially on lossy networks.

What is CORS and how does it work? +

CORS (Cross-Origin Resource Sharing) is a security mechanism that allows web applications running at one origin to access resources from a different origin. Browsers enforce the same-origin policy by default, which prevents web pages from making requests to a different domain than the one that served the web page.

CORS works by using additional HTTP headers that tell browsers to give a web application running at one origin, access to selected resources from a different origin.

What is Content Security Policy (CSP)? +

Content Security Policy (CSP) is a security standard that helps prevent cross-site scripting (XSS), clickjacking, and other code injection attacks. It allows website administrators to specify which domains the browser should consider as valid sources of executable scripts, styles, images, and other resources.

CSP is implemented via HTTP headers or meta tags and works by whitelisting sources of approved content, preventing the browser from loading malicious assets.

What are WebSockets and how do they differ from HTTP? +

WebSockets provide full-duplex communication channels over a single TCP connection, allowing real-time data exchange between client and server. Key differences from HTTP:

  • Connection: WebSockets maintain a persistent connection, while HTTP creates new connections for each request
  • Communication: WebSockets enable bidirectional communication, while HTTP is primarily request-response
  • Overhead: WebSockets have lower overhead after the initial handshake
  • Use cases: WebSockets are ideal for real-time applications like chat, gaming, and live updates
What is the difference between cookies, localStorage, and sessionStorage? +

Cookies: Small pieces of data sent from a website and stored on the user's computer. They are sent with every HTTP request to the domain, have size limits (~4KB), and can have expiration dates.

localStorage: Web storage API that allows JavaScript to store data in the browser with no expiration time. Data persists until explicitly deleted, with larger storage limits (typically 5-10MB).

sessionStorage: Similar to localStorage but data is cleared when the page session ends (when the browser is closed). Useful for temporary data that shouldn't persist across sessions.

Security Questions

What is HTTPS and how does it work? +

HTTPS (Hypertext Transfer Protocol Secure) is the secure version of HTTP that uses encryption to protect data in transit. It works using TLS (Transport Layer Security) or its predecessor SSL (Secure Sockets Layer) to encrypt communications between web browsers and servers.

The HTTPS process involves:
1. TCP connection establishment
2. TLS handshake to establish encryption parameters
3. Certificate verification to ensure server authenticity
4. Encrypted data exchange using symmetric cryptography

What are SSL/TLS certificates and how are they verified? +

SSL/TLS certificates are digital documents that bind a cryptographic key to an organization's details. They are used to authenticate the identity of a website and enable encrypted connections.

Certificate verification involves:
1. Checking the certificate's validity period
2. Verifying the certificate signature against a trusted Certificate Authority (CA)
3. Ensuring the certificate matches the requested domain name
4. Checking for certificate revocation through CRL or OCSP

What is two-factor authentication (2FA) and how does it work? +

Two-factor authentication (2FA) is a security process that requires users to provide two different authentication factors to verify their identity. These factors typically fall into three categories:

  • Knowledge - Something you know (password, PIN)
  • Possession - Something you have (phone, security key)
  • Inherence - Something you are (fingerprint, facial recognition)

2FA works by requiring a combination of two factors, making it significantly harder for attackers to gain unauthorized access even if they compromise one factor (like a password).

What is a CDN and how does it improve security? +

A CDN (Content Delivery Network) is a geographically distributed network of proxy servers and their data centers that helps deliver content efficiently to users. CDNs improve security by:

  • DDoS protection - Absorbing and mitigating distributed denial-of-service attacks
  • Web application firewall (WAF) - Filtering and monitoring HTTP traffic
  • SSL/TLS termination - Handling encryption/decryption at the edge
  • Bot mitigation - Identifying and blocking malicious bots
  • Obfuscating origin server IP - Protecting the actual server from direct attacks
What is the same-origin policy and why is it important? +

The same-origin policy is a critical security mechanism that restricts how a document or script loaded from one origin can interact with a resource from another origin. Two URLs have the same origin if they have the same protocol, host, and port.

This policy is important because it prevents malicious scripts from one origin from accessing sensitive data from another origin. It helps isolate potentially malicious documents, reducing possible attack vectors.

Cross-origin interactions are possible through mechanisms like CORS, but they require explicit permission from the server.

Performance Questions

What factors affect website loading speed? +

Website loading speed is affected by numerous factors including:

  • Network factors: Latency, bandwidth, packet loss
  • Server factors: Server processing power, database queries, application logic
  • Content factors: Number of requests, file sizes, compression
  • Client factors: Browser capabilities, device performance, cache status
  • DNS resolution time: Time to resolve domain names to IP addresses
  • SSL/TLS negotiation: Time to establish secure connections
  • Render blocking resources: CSS and JavaScript that delay page rendering
How does browser caching work and how can it be optimized? +

Browser caching stores web resources locally so they can be reused instead of re-downloaded on subsequent visits. Optimization strategies include:

  • Setting appropriate Cache-Control headers with max-age directives
  • Using ETags for validation of unchanged resources
  • Implementing versioning or fingerprinting for static assets
  • Leveraging CDNs for edge caching
  • Using service workers for programmatic caching control
  • Setting different caching policies for different resource types
What is lazy loading and how does it improve performance? +

Lazy loading is a technique that delays the loading of non-critical resources until they are needed. Instead of loading all resources when the page first loads, lazy loading defers loading of below-the-fold content or content that might not be needed immediately.

Benefits include:
• Reduced initial page load time
• Lower bandwidth consumption
• Reduced system resource usage
• Improved perceived performance

Modern browsers support native lazy loading for images and iframes using the loading="lazy" attribute.

What are critical rendering path optimizations? +

The critical rendering path is the sequence of steps the browser goes through to convert HTML, CSS, and JavaScript into pixels on the screen. Optimizations include:

  • Minifying and compressing resources
  • Eliminating render-blocking resources
  • Optimizing CSS delivery (inline critical CSS)
  • Deferring non-critical JavaScript
  • Preloading important resources
  • Reducing the number of critical resources
  • Minimizing the length of the critical path
